# TINC Product Philosophy

## Purpose

TINC exists to help people transform an electronic idea into a real physical product.

Not just a schematic.

Not just a PCB.

Not just a simulation.

A complete manufacturable product.

------------------------------------------------------------

## What TINC Is

TINC is NOT an IDE.

TINC is NOT a programming environment.

TINC is NOT a CAD clone.

TINC is NOT a KiCad clone.

TINC is NOT a Wokwi clone.

TINC is NOT a Visio clone.

TINC is a Hardware Product Design Environment.

The goal is to help users design, organize, understand and manufacture real electronic products.

------------------------------------------------------------

## Product First

Engineering exists to serve the product.

The product does not exist to showcase engineering.

Whenever there is a conflict between engineering convenience and user experience,

always prioritize user experience.

------------------------------------------------------------

## The Interface Is Invisible

Users should think about hardware.

Never about the software.

The interface should disappear behind the work.

Every visible UI element must justify its existence.

------------------------------------------------------------

## Remove Before Adding

Before adding a button,

remove another.

Before adding a menu,

simplify another.

Before adding a workflow,

simplify an existing workflow.

Complexity must only increase when it provides significant user value.

------------------------------------------------------------

## Discoverability Before Memorization

Users should never be required to memorize commands.

The interface should naturally reveal what can be done next.

Shortcuts accelerate workflows.

They never replace discoverability.

------------------------------------------------------------

## Reality Before Abstraction

Users think in products.

Not entities.

Not nodes.

Not JSON.

Not symbols.

Not implementation objects.

Users think:

"I'm building a weather station."

"I'm building a robot."

"I'm building a smart lock."

The interface should communicate using the user's mental model.

Never the internal implementation model.

------------------------------------------------------------

## Direct Manipulation

Users should manipulate objects directly whenever possible.

Drag.

Drop.

Move.

Rotate.

Resize.

Connect.

Avoid unnecessary dialogs.

Avoid property windows for simple actions.

------------------------------------------------------------

## Progressive Disclosure

Beginners should never feel overwhelmed.

Experts should never feel limited.

Simple workflows should remain simple.

Advanced functionality should appear only when needed.

------------------------------------------------------------

## Physical First

Every design decision should move users closer to the real object they will eventually build.

The software should constantly reinforce the relationship between the digital design and the physical product.

------------------------------------------------------------

## WYSIWYG

The design should visually resemble the real hardware as closely as practical.

Users should recognize their future physical product from what they see on screen.

------------------------------------------------------------

## Native First

External assets are supported.

Imported assets are welcome.

Native TINC assets define the product identity.

The TINC Visual Component Standard is always the reference implementation.

------------------------------------------------------------

## Consistency Over Features

A smaller, coherent product is better than a larger inconsistent one.

Never sacrifice consistency simply to add another feature.

------------------------------------------------------------

## Design Language

The interface should feel calm.

Simple.

Modern.

Professional.

Confident.

Avoid visual noise.

Avoid decorative UI.

Avoid duplicated controls.

Prefer whitespace over borders.

Prefer hierarchy over color.

Prefer typography over decoration.

------------------------------------------------------------

## Workspace Philosophy

TINC is not software.

TINC is a digital electronics workshop.

Every workspace should feel like a real engineering bench.

Users are not editing files.

Users are building products.

------------------------------------------------------------

## Long-Term Vision

The complete TINC workflow is:

Design

↓

Build

↓

Understand

↓

Physical Layout

↓

Mechanical Integration

↓

Manufacturing

Every future workspace should naturally fit somewhere inside this workflow.

No feature should exist in isolation.

------------------------------------------------------------

## Decision Rule

Before implementing anything, ask:

Does this make the product simpler?

Does this help a first-time user?

Does this reduce friction?

Does this remove complexity?

Does this move the user closer to building a real product?

If the answer is "No",

do not implement it.
